Ce document constitue la spécification fonctionnelle complète de skilo. Il détaille l'ensemble des fonctionnalités Must Have avec leurs règles métier, préconditions, postconditions, cas d'erreur et exemples concrets. Il sert de référence unique pour le développement.

# 1\. Présentation du projet

## 1.1 Concept

**skilo** est une plateforme web permettant à n'importe quel individu d'échanger ses compétences avec une autre personne - sans argent, sans intermédiaire, uniquement du temps et du savoir. La promesse fondamentale : "Tu m'apprends ce que tu sais, je t'apprends ce que je sais."

La plateforme introduit également un système de crédits temps permettant de débloquer des apprentissages même en l'absence de match mutuel direct, inspiré du modèle des banques de temps communautaires.

## 1.2 Problème résolu

| **Problème**                               | **Solution skilo**                                                    |
| ------------------------------------------ | ------------------------------------------------------------------------- |
| Les cours en ligne coûtent cher            | 100% gratuit - échange de valeur sans argent                              |
| L'IA explique mais ne pratique pas         | Interaction humaine réelle, tutorat personnalisé                          |
| Les profils de ton entourage sont inconnus | Matching algorithmique basé sur les compétences déclarées                 |
| Pas de match mutuel disponible             | Système de crédits temps pour contourner l'absence de réciprocité directe |

## 1.3 Utilisateurs cibles

- L'étudiant en code qui veut apprendre l'anglais
- Le graphiste qui veut apprendre Excel
- Le musicien qui veut apprendre la photo
- Le comptable qui veut apprendre à parler en public

Tout le monde a quelque chose à offrir. Tout le monde a quelque chose à apprendre.

### FC-03-A : Calcul des matchs

<div class="joplin-table-wrapper"><table><tbody><tr><th><p><strong>Identifiant</strong></p></th><th><p>FC-03-A</p></th></tr><tr><td><p><strong>Objectif</strong></p></td><td><p>Identifier automatiquement les utilisateurs avec qui un échange de <strong>compétences</strong> est possible, en distinguant <strong>matchs parfaits</strong> et <strong>matchs partiels</strong>.</p></td></tr><tr><td><p><strong>Préconditions</strong></p></td><td><ul><li>L'utilisateur a au moins 1 compétence offerte ET 1 compétence recherchée.</li><li>Le recalcul est déclenché : à la connexion, après modification du profil, selon un job planifié toutes les heures.</li></ul></td></tr><tr><td><p><strong>Définition - Match parfait</strong></p></td><td><ul><li>Il existe au moins une <strong>paire (X, Y)</strong> telle que :</li><li>→ <strong>Utilisateur A</strong> offre <strong>X</strong> (principale ou secondaire) ET cherche <strong>Y</strong></li><li>→ <strong>Utilisateur B</strong> offre <strong>Y</strong> (principale ou secondaire) ET cherche <strong>X</strong></li><li>Exemple : A offre Figma + Python, cherche React. B offre React, cherche Python. → <strong>Match parfait sur la paire (Python ↔ React).</strong></li></ul></td></tr><tr><td><p><strong>Définition - Match partiel</strong></p></td><td><ul><li>B offre au moins une compétence <strong>que A cherche</strong>, MAIS B <strong>ne cherche rien que A offre</strong>.</li><li>La relation est à sens unique sur le plan de la réciprocité.</li><li>Exemple : A cherche React. B offre React mais cherche Swift que A n'offre pas. →<strong> Match partiel.</strong></li></ul></td></tr><tr><td><p><strong>Calcul du score de compatibilité - Match parfait</strong></p></td><td><ul><li>Base : <strong>+50 points</strong> par paire de compétences mutuelles trouvée.</li><li>Bonus niveau : <strong>+20 points</strong> si le niveau offert correspond exactement au niveau recherché.</li><li>Bonus niveau partiel : <strong>+10 points</strong> si le niveau offert est supérieur d'un cran au niveau recherché.</li><li>Seuils d'affichage : 70-100 = "Très compatible", 50-69 = "Compatible", 20-49 = "Partiellement compatible".</li></ul></td></tr><tr><td><p><strong>Calcul du score - Match partiel</strong></p></td><td><ul><li>Score basé uniquement sur la correspondance dans le sens qui existe.</li><li>Base : +40 points par compétence que B offre et que A cherche.</li><li>Bonus niveau : +15 points si le niveau correspond exactement.</li><li>Pas de bonus réciprocité.</li></ul></td></tr><tr><td><p><strong>Logique de non-répétition</strong></p></td><td><ul><li>Un utilisateur <strong>déjà en session active</strong> (<strong>pending</strong> ou <strong>confirmed</strong>) avec A n'apparaît pas dans la liste des matchs.</li><li>Il réapparaît après que la session est passée en <strong>completed</strong> ou <strong>cancelled</strong>.</li></ul></td></tr><tr><td><p><strong>Postconditions</strong></p></td><td><ul><li>Les matchs sont stockés en table <strong>match</strong> avec <strong>score</strong>, <strong>type (perfect/partial</strong>), <strong>status</strong>.</li><li>La liste est <strong>rechargée</strong> côté front après <strong>chaque recalcul</strong>.</li><li>Une notification est envoyée si un nouveau match parfait apparaît.</li></ul></td></tr></tbody></table></div>

### FC-04-A : Proposition d'une session

### FC-04-B : Cycle de vie d'une session

Une session passe par les statuts suivants :

| **Statut**     | **Déclencheur**                            | **Description**                                         |
| -------------- | ------------------------------------------ | ------------------------------------------------------- |
| pending        | Proposition envoyée                        | En attente de réponse du destinataire                   |
| confirmed      | Destinataire accepte                       | Les deux parties sont engagées                          |
| completed      | Les deux confirment la tenue               | Échange réalisé - évaluation débloquée                  |
| cancelled      | L'un ou l'autre annule                     | Session annulée avant ou après confirmation             |
| auto-completed | 1 confirme, l'autre ne répond pas sous 24h | Complétion automatique pour ne pas bloquer l'évaluation |

<div class="joplin-table-wrapper"><table><tbody><tr><th><p><strong>Identifiant</strong></p></th><th><p>FC-04-B</p></th></tr><tr><td><p><strong>Acceptation (pending → confirmed)</strong></p></td><td><ul><li>Le destinataire reçoit la notification et accède au détail de la session.</li><li>Il peut accepter (confirmed) ou refuser (cancelled avec motif optionnel).</li><li>En cas d'acceptation : l'initiateur est notifié : "[Prénom] a accepté votre session du [date]."</li><li>En cas de refus : l'initiateur est notifié : "[Prénom] a décliné votre session." + motif si renseigné.</li></ul></td></tr><tr><td><p><strong>Confirmation de complétion (confirmed → completed)</strong></p></td><td><ul><li>Après la date/heure de la session, les deux utilisateurs reçoivent : "Votre session avec [Prénom] a-t-elle eu lieu ?"</li><li>Chacun confirme indépendamment (boutons Oui / Non).</li><li>Les deux confirment Oui → statut completed → évaluation débloquée.</li><li>L'un dit Non → session passée en litige (statut disputed) → les deux sont notifiés.</li><li>L'un confirme, l'autre ne répond pas dans 24h → statut auto-completed → évaluation débloquée.</li><li>Les deux disent Non → statut cancelled.</li></ul></td></tr><tr><td><p><strong>Rappel automatique</strong></p></td><td><ul><li>1 heure avant la session confirmée : notification in-app aux deux utilisateurs.</li><li>Contenu : "Rappel - Votre session avec [Prénom] commence dans 1 heure. [Lien si en ligne]"</li></ul></td></tr><tr><td><p><strong>Annulation</strong></p></td><td><ul><li>Possible par l'initiateur ou le destinataire à tout moment avant completed.</li><li>L'autre partie est notifiée immédiatement.</li><li>Motif d'annulation optionnel (max 200 caractères).</li><li>Si annulation &lt; 2h avant la session confirmée : avertissement affiché (impact potentiel sur la réputation - fonctionnalité de réputation à implémenter en Should Have).</li></ul></td></tr><tr><td><p><strong>Historique des sessions</strong></p></td><td><ul><li>Vue "Mes sessions" avec deux onglets : À venir / Passées.</li><li>Chaque session affiche : photo + prénom de l'autre, date, durée, compétences échangées, statut (couleur + libellé).</li><li>Statuts visuels : pending = gris, confirmed = bleu, completed = vert, cancelled = rouge, disputed = orange.</li></ul></td></tr></tbody></table></div>

### FC-06-A : Principe et modèle

Le système de crédits temps est la solution skilo au problème d'absence de réciprocité directe. Inspiré des banques de temps communautaires, il permet de monétiser le savoir sans argent, en utilisant le temps comme monnaie d'échange universelle.

Règle fondamentale : 1 heure enseignée = 1 crédit temps gagné. 1 crédit temps = accès à 1 heure d'apprentissage auprès de n'importe quel utilisateur, même sans match mutuel.

<div class="joplin-table-wrapper"><table><tbody><tr><th><p><strong>Identifiant</strong></p></th><th><p>FC-06-A</p></th></tr><tr><td><p><strong>Sources de crédits</strong></p></td><td><ul><li>Crédit de bienvenue : 2 crédits attribués à l'inscription (encourage le premier échange).</li><li>Session complétée en tant qu'enseignant : +1 crédit par heure enseignée (arrondi à l'heure supérieure).</li><li>Exemple : enseigner 1h30 → +2 crédits. Enseigner 45min → +1 crédit.</li><li>Bonus "Profil complet" : +1 crédit unique attribué quand l'indicateur de force du profil atteint 100%.</li></ul></td></tr><tr><td><p><strong>Utilisation des crédits</strong></p></td><td><ul><li>Un crédit permet de proposer une session à un utilisateur avec qui on a un match partiel ou aucun match direct.</li><li>Pour un match parfait : aucun crédit requis - l'échange est naturellement réciproque.</li><li>Coût : 1 crédit par heure de session (une session d'1h30 coûte 2 crédits).</li><li>Les crédits sont débités au moment de l'acceptation de la session (pas à la proposition).</li><li>Si la session est annulée avant completed : les crédits sont remboursés intégralement.</li></ul></td></tr><tr><td><p><strong>Règles de gestion</strong></p></td><td><ul><li>Le solde de crédits ne peut jamais être négatif.</li><li>Maximum de crédits accumulables : 20 crédits (prévention du stockage excessif sans échange).</li><li>Le solde est visible uniquement par l'utilisateur lui-même - # 3\. Règles transversales

## 3.1 Sécurité

- Toutes les routes authentifiées requièrent un access token JWT valide dans le header Authorization: Bearer &lt;token&gt;.
- La validation des données est effectuée côté back-end (class-validator NestJS) indépendamment de la validation front.
- Aucune donnée sensible (mot de passe, token, email d'un tiers) n'est retournée dans les réponses API.
- Les IDs en base sont des UUID v4 - jamais des entiers auto-incrémentés exposés dans les URLs.
- Les uploads de fichiers sont validés en type MIME côté serveur - pas seulement en extension.

## 3.2 Performance

- Toutes les listes (matchs, sessions, évaluations) sont paginées (défaut : 20 items par page).
- Les champs fréquemment filtrés sont indexés en base : compétences (nom, catégorie), sessions (statut, date), users (email).
- Le recalcul du matching est asynchrone - il ne bloque jamais la réponse à l'utilisateur.

## 3.3 Codes de retour API

| **Code**                  | **Signification**      | **Exemple d'usage**             |
| ------------------------- | ---------------------- | ------------------------------- |
| 200 OK                    | Succès                 | GET profil, GET matchs          |
| 201 Created               | Ressource créée        | POST session, POST inscription  |
| 400 Bad Request           | Données invalides      | Formulaire mal rempli           |
| 401 Unauthorized          | Token absent ou expiré | Accès route protégée sans token |
| 403 Forbidden             | Accès refusé           | Modifier le profil d'un autre   |
| 404 Not Found             | Ressource introuvable  | Session inexistante             |
| 409 Conflict              | Conflit de données     | Email déjà utilisé              |
| 500 Internal Server Error | Erreur serveur         | Erreur inattendue               |
